# Architecture

The project should be data-first and reader-driven.

## Intended Flow

```text
vendor Standard Ebooks XHTML
  -> ingestion scripts
  -> chapter manifest and normalized text data
  -> summaries, glossary, reference cards, reading paths, annotations
  -> selector refresh and generated indexes
  -> schema validation and custom audits
  -> Astro reader site
```

## Core Principle

The full text is the base layer. Everything else is metadata or display logic.

Shorter editions should be generated by reading-path definitions that include, excerpt, summarize, or defer chapters. No pipeline step should permanently remove source text.

## Proposed Top-Level Structure

| Path | Purpose |
|---|---|
| `vendor/standard-ebooks/` | Upstream source, preserved and read-only. |
| `data/chapters/` | Chapter manifest, summaries, tags, and reading-path inclusion flags. |
| `data/glossary/` | Terms and student-facing definitions. |
| `data/references/` | Source-cited reference cards. |
| `data/indexes/` | Generated annotation lookup, tag, density, and edge indexes. |
| `data/paths/` | Reading path definitions. |
| `data/taxonomy/` | Controlled trail and entity IDs for future semantic indexes. |
| `data/geography/` | Place and route-stage records with coordinate status and source notes. |
| `schemas/` | JSON Schemas for validation. |
| `scripts/ingest/` | Source extraction and normalization. |
| `scripts/validate/` | Data audits. |
| `src/` | Astro reader app, components, styles, and data-loading helpers. |
| `dist/` | Generated Astro build output. Do not hand-edit. |

## Reader Shape

The first screen of the future site should be the usable reading experience: path selector, chapter navigation, current text, and contextual layers. A landing page can come later if publication needs one.

Astro is the required platform for the final product. The first page should remain the reader itself, with reusable components and content collections added when they reduce complexity or strengthen validation.

Do not migrate the reader wholesale to React before the content scale-up. Astro should stay the shell for static text, indexed data, and fast page output. React can be introduced later as isolated Astro islands for genuinely stateful visualization surfaces: faceted search, graph/network exploration, map filtering, annotation-density heatmaps, and teacher review dashboards.
